# Phase 17 — UI Design Contract

> Visual and interaction contract for frontend phases. Generated by gsd-ui-researcher, verified by gsd-ui-checker.

---

## Scope of This Contract

Phase 17 ships a **second renderer** (`WebGLConstellation`) plugged into the **identical Phase 15 props contract** as `SvgConstellation`. This is **not a redesign** — it is the same visual identity rendered through a different graphics layer, plus a small set of **net-new motion behaviors** that exploit WebGL's per-frame draw budget.

### What this contract INHERITS without modification

Every visual concern below is **locked by Phase 15 + Phase 16** and re-asserted as a binding parity contract for the WebGL renderer. The WebGL renderer MUST visually match the SVG renderer at parity zoom — a recruiter switching renderers (via `?renderer=svg|webgl` override) MUST NOT see a different static design.

### What this contract ADDS

Five small, contained additions specific to the WebGL render path:

1. **Ambient drift** — gentle 2D per-node positional perturbation while idle.
2. **Glow pulse** — selected node radius/alpha pulse on a 2s loop.
3. **Halo brighten** — animated alpha lift on highlighted-skill node halos.
4. **Chip-flash shader uniform** — translates Phase 16's `chipFlash` CSS keyframe into a shader-side scale/alpha pulse on `justFilteredId`.
5. **Renderer-swap / fallback / loading visuals** — instant snap (no crossfade) on capability change OR ErrorBoundary fallback OR Suspense chunk fetch.

### Theme Reactivity (Claude's Discretion: D-17-THEME)

**Recommended approach (planner finalizes):** Single-source theme prop pass-through.

| Mechanism | Trade-off | Recommendation |
|-----------|-----------|----------------|
| Option A: `useTheme()` hook → `theme` prop → effect re-reads CSS vars on prop change → updates shader uniforms imperatively | Aligns with Phase 15 contract (`theme` already in props). Zero new event channels. Theme change → React re-render → `useEffect([theme])` triggers uniform update. | **PREFERRED** |
| Option B: `MutationObserver` on `<html data-theme>` attribute → callback re-reads vars → updates uniforms | Decouples renderer from React tree; works even if `theme` prop is stale. But adds an observer + duplicates the React-internal theme-change signal. | Acceptable fallback only if Option A reveals a race in testing |

**Contract:** when theme changes (dark ↔ light), the renderer MUST update `uEdgeColor`, `uEdgeHeavyColor`, `uHaloColor`, `uHaloSecondary`, AND the per-vertex `uNodeStrokeColor[i]` attribute (re-uploaded to GPU OR a uniform array). Node fill colors (`uNodeColor[i]`) do NOT change with theme (data values). Update timing: within one rAF frame of the theme-change event; no visible "wrong-theme flash" tolerated.

### Accent reserved for (combined list across phases)

Inherits Phase 15 + Phase 16 lists. Phase 17 adds **one** reserved use:

10. WebGL renderer's selected-node halo accent ring (rendered in shader as a thin band at `r ≤ d < r + 2` SVG units when a node is `selectedSkillId`, brand color). This mirrors the Phase 15 `<circle stroke="var(--color-brand)" strokeWidth="2">` ring that appears on the focused node — WebGL reproduces it for the selected state (focus ring stays a DOM concern; see §"Focus indicator on WebGL").

Accent is **NOT** used for: node fills, edge strokes, ambient drift, glow pulse, halo brighten.

### WCAG AA Contrast verification

Inherited verbatim from Phase 15 §"WCAG AA Contrast verification". Phase 17 introduces **zero new color pairs**. The WebGL renderer produces the same visible nodes/edges/halos against the same backdrop (`ink-800` dark `#1A1A2E` / light `#F0F0F5`); contrast ratios from Phase 15 apply unchanged.

Particular care: the light-theme stroke ring (1.5-equivalent-px in `LIGHT_THEME_STROKES` colors) MUST be visible at WebGL render. Anti-aliasing in the fragment shader must not blur the stroke below the 3:1 UI-component threshold. **Acceptance check during executor visual UAT:** screenshot diff between SVG and WebGL renderer at light theme, light-theme stroke ring must read as a visible ring band (≥1.5 effective device px after AA) on every node.

## Active-State Contract (parity with Phase 15/16)

| State | Visual treatment (parity rule) | WebGL implementation note |
|-------|-------------------------------|---------------------------|
| Resting (no selection, no filter) | All nodes `fill-opacity: 1.0`; weight ≥2 edges visible at full opacity; weight-1 edges hidden | Per-vertex `dim = 0`, `halo = 0`, `glow = 0`. Edge shader hides weight-1 unless `uActiveNodeId === source OR target`. |
| Hover (no selection yet) | Hovered node halo at single-pass dim glow (`drop-shadow(0 0 4px halo)`); incident weight-1 edges fade in | WebGL: shader uniform `uHoveredNodeId`; halo intensity `0.4`; weight-1 incident edges fade in via uniform `uEdgeRevealId`. |
| Selected | Selected node halo at two-pass full glow; brand accent ring (see §Color above); non-selected nodes dim to `fill-opacity: 0.35` IFF a filter is NOT active | WebGL: per-vertex `dim` attribute re-uploaded on selection change. Halo on selected via fragment additive blend. |
| Filter highlight (Phase 16 `highlightedSkillIds`) | Nodes in `highlightedSkillIds` at 1.0; others at 0.35 | WebGL: per-vertex `dim` attribute reflects `shouldDimNode` result (same predicate as `SvgConstellation.js` lines 84–95 — port verbatim) |
| Filter year-range mismatch | Same dim treatment (0.35) | Same `dim` attribute |
| Edge incident to dimmed node | Edge opacity multiplied by 0.35 (Phase 15 line 233) | WebGL: per-segment edge shader multiplies by `min(node1.dim, node2.dim) × 0.35 + (1 − min) × 1.0` |
| Chip-flash (`justFilteredId`) | 100ms `chipFlash` keyframe: opacity 1 → 0.5 → 1; scale 1 → 0.94 → 1 | WebGL: shader uniform `uFlashNodeId` + `uFlashStartTime`; fragment scales `gl_PointSize` and alpha over 100ms; see §"Chip-Flash on WebGL" |

**Initial resting state rule (re-asserted from Phase 15):** all nodes `fill-opacity: 1.0` on mount; the 0.35 dim is a POST-SELECTION/POST-FILTER state, never a default. The WebGL renderer's `dim` attribute MUST be initialized to 0 on first frame.

## Sizing Scale — WebGL primitives

### Node geometry

WebGL nodes are `three.Points` with custom `ShaderMaterial`. Per-vertex `BufferAttribute`s for size, color, glow intensity, halo factor, dim factor, and (light-theme) stroke color.

| Property | Value | Parity rule |
|----------|-------|-------------|
| Node radius (logical SVG units) | Same formula as Phase 15: `floor + (√count / √MAX_COUNT) × (ceil − floor)`; desktop floor 10 / ceil 23 | Identical numerical output to SVG renderer at every node |
| `gl_PointSize` mapping | `gl_PointSize = svgRadius × pixelRatio × cameraScale × 2` (the ×2 because `gl_PointSize` is the diameter of the point sprite, not radius) | Tuned so a 10-SVG-unit radius node measures ~16 device px on a 1× display at the desktop 600px canvas height — matches Phase 15 desktop visual radius (~8–18 device px range) |
| DPR awareness | `renderer.setPixelRatio(Math.min(window.devicePixelRatio, 2))` | Clamps Retina at 2× to avoid 4×-pixel-count cost on 4K displays; matches industry default |
| Touch target | N/A on WebGL (desktop-only path; mouse hover + keyboard nav from DOM wrapper) | The 22-SVG-unit transparent hit circle from Phase 15 SVG path is NOT needed in WebGL — pointer events are captured at the `<canvas>` level and ray-picked in JS |
| Pointer pick | Screen-space distance test from mouse position to each projected node center; pick closest within `pickRadius = currentSvgRadius + 8` device px | Slightly larger than visual radius to make hover forgiving on small nodes; matches the SVG path's intent (which used a 22-unit hit circle) |

### Edge geometry

WebGL edges are `three.LineSegments` with `BufferGeometry` (2 vertices per edge) + a per-segment shader uniform OR a per-vertex attribute carrying the edge weight + reveal state.

| Property | Value | Parity rule |
|----------|-------|-------------|
| Stroke width | Same formula as Phase 15: `weight === 1 ? 1.0 : min(1.0 + (w−1) × 1.0, 4.0)` SVG units → multiplied by `pixelRatio × cameraScale` for device px | Note: WebGL `LineBasicMaterial.linewidth` is broken in WebGL2 (always 1px on most drivers). Two acceptable paths: (a) accept 1px lines (visual difference vs SVG; rejected — fails parity), (b) use `three-meshline` ~6 kB gz OR custom `BufferGeometry` quad strips for thick lines. **Planner decision required.** Recommendation: custom quad-strip BufferGeometry — zero new deps, ~20 LOC, parity-perfect with SVG. |
| Edge color | `uEdgeColor` for weight 1; `uEdgeHeavyColor` for weight ≥2 | Same hex resolution as Phase 15 reads `var(--color-constellation-edge*)` |
| Weight-1 reveal | Hidden by default (alpha 0); shows when incident to `uActiveNodeId` (selected OR focused OR hovered, fall-through order) | Per-segment alpha computed in vertex/fragment shader from `uActiveNodeId` uniform |
| Edge dim on filter | Multiply alpha by 0.35 when either endpoint node is dimmed (`min(source.dim, target.dim) > 0`) | Mirrors Phase 15 `SvgConstellation.js` lines 227–234 |
| Single draw call | All ~50 edges in one `LineSegments` mesh; all visibility/dim/reveal logic shader-side | D-17-PRIMITIVES |

### Halo (selected-node outer glow)

The Phase 15 SVG renderer applies halo via CSS `filter: drop-shadow(0 0 8px halo) drop-shadow(0 0 16px rgba(255,255,255,0.08))`. WebGL reproduces this in the point-sprite fragment shader as an additive radial gradient.

| Property | Value | Implementation hint |
|----------|-------|---------------------|
| Inner glow radius | 8 SVG units beyond the node fill edge | Fragment shader: `additiveAlpha = uHaloColor.a × smoothstep(r + 8, r + 0, dist)` for the inner pass |
| Outer glow radius | 16 SVG units beyond the node fill edge | Second fragment pass with `uHaloSecondary` color |
| Selected-node treatment | Both passes active; total halo extent ~16 units beyond fill | `gl_PointSize` must be wide enough to contain the halo — increase sprite quad size to `radius + 16 + 2px AA buffer` so the gradient is not clipped |
| Hover (non-selected) treatment | Inner pass only at half intensity (`alpha × 0.5`); no outer pass | Matches Phase 15 "Hover (non-active)" row |
| Highlighted (Phase 16 `highlightedSkillIds`) treatment | Net-new in Phase 17: halo brighten — see §"Halo brighten on highlighted" below |
| Reduced-motion path | N/A — WebGL never mounts under reduced-motion (D-17-CAP-GATES) | No branch needed in WebGL fragment shader |

## Animation Contract

Phase 17 introduces FIVE new shader-driven animations. All have explicit budgets and respect Phase 17's "wow but not jarring" boundary. Reduced-motion is NOT a concern here: D-17-CAP-GATES already routes reduced-motion users to the SVG path before WebGL ever mounts.

### 1. Ambient drift (idle motion)

Continuous gentle per-node perturbation around the baked layout position, signaling "alive" without distracting from selection.

| Property | Value | Rationale |
|----------|-------|-----------|
| Direction | 2D (x AND y) per node — each node has its own random phase + frequency seed | Pure horizontal drift would read as a global slide; 2D with per-node seeds reads as twinkle |
| Amplitude | 0.2 SVG units (≈0.16 device px at 800px canvas width on 1× display; ≈0.32 device px at 2× DPR) | Sub-perceptual at rest — feels "alive" rather than "moving" |
| Period | 4–6 seconds per axis per node (random seeded value in `[4000, 6000] ms`) | Slow enough to never trigger motion-sickness; fast enough to be perceived |
| Curve | Pure sin: `pos = baked + amp × sin(2π × t / period + phase)` | Cheapest GLSL primitive; analytically smooth; no perceptual judder |
| Per-node deterministic seed | `hash(node.id)` → `[phase, periodX, periodY]` | Same node drifts identically across reloads; visual consistency for QA screenshots |
| Pause | rAF loop pauses on `document.visibilityState === 'hidden'` (D-17-FRAMELOOP) | Battery + CPU saving; resumes seamlessly on visible |
| Suspends on | Never — always-on while WebGL is mounted. Selection and filter changes do NOT pause drift. | Constant ambient life |
| **Boundary check** | If amplitude × scale ever exceeds 1.0 device px AT ANY DPR, REDUCE to fit. Drift must NEVER be the reason a recruiter notices "movement." | |

### 2. Glow pulse (selected node)

The selected node breathes its halo: radius scale + alpha modulation on a 2s loop.

| Property | Value | Rationale |
|----------|-------|-----------|
| Trigger | `selectedSkillId !== null && uActiveNodeId === selectedSkillId` (any node currently the selection target) | Single-node visual anchor — recruiter's eye stays on the answer |
| Halo radius scale | `1.0 → 1.15 → 1.0` over 2000 ms, ease-in-out (sinusoidal: `0.5 + 0.5 × cos(2π × t / 2000)` mapped to `[1.0, 1.15]`) | 15% radius lift is enough to read as a "pulse" without growing the perceived node size on the page |
| Halo alpha scale | `1.0 → 1.2 → 1.0` over 2000 ms, ease-in-out (same sin curve, mapped to `[1.0, 1.2]`) — clamped so result never exceeds `uHaloColor.a × 1.0` (i.e., the halo MAX is what Phase 15 calls "selected halo") | Halo BREATHES; it does not exceed the static brightness Phase 15 ships |
| Period | 2000 ms (matches Phase 15 `pulse2` cadence on the biggest-node pre-interaction hint) | Same rhythm the user already saw on initial pulse — recognizable language |
| Node body | Not affected — only the halo modulates | Node fill color/size stays stable so position reads remain accurate |
| Suspends on | Selection cleared (`selectedSkillId === null`) → pulse stops at next frame, halo collapses to 0 over 200 ms ease-out | No popping; gentle return to rest |

### 3. Halo brighten on highlighted-skill IDs

When Phase 16 filters produce `highlightedSkillIds`, those nodes glow softly (subtler than the selected pulse) to reinforce the dim/highlight visual contrast WebGL ALREADY provides via `dim` attribute.

| Property | Value | Rationale |
|----------|-------|-----------|
| Trigger | `highlightedSkillIds.includes(node.id)` for any node NOT the selected node | Selected node owns the pulse animation; highlighted nodes get the brighten layer |
| Halo alpha | `0.4 × uHaloColor.a → 0.8 × uHaloColor.a → 0.4` over 3000 ms, ease-in-out | Slower (3s) than the selected pulse (2s) so the eye distinguishes "selected = primary action" vs "highlighted = also relevant" |
| Halo radius | Fixed at `+4 SVG units` (no breath) | Stable visual mass for highlighted nodes; only alpha varies |
| Period | 3000 ms | Distinct from the 2000 ms selected-node pulse |
| Per-node phase offset | `hash(node.id) × 3000` ms | Highlighted nodes breathe out of sync — feels organic, not strobed |
| Suspends on | `highlightedSkillIds` empty OR node leaves the highlight set → fade to alpha 0 over 200 ms | Clean exit |

### 4. Chip-flash on WebGL (parity with Phase 16 `chipFlash` keyframe)

When `justFilteredId` prop changes (per Phase 16 D-16-CHIP-FLASH — the user clicked a tech chip and the matching node MUST flash), the WebGL renderer reproduces the 100ms `chipFlash` keyframe.

| Property | Value | Parity rule |
|----------|-------|-------------|
| Trigger | `justFilteredId === node.id` for one node | Single shader uniform `uFlashNodeId` + `uFlashStartTime` set when prop changes |
| Duration | 100 ms total (matches Phase 16 keyframe `chipFlash: 0% → 50% → 100%` over 100ms ease-out) | Single shared timing budget |
| `gl_PointSize` scale curve | `1.0 → 0.94 → 1.0` (matches Phase 16 transform: scale curve), ease-out (cubic: `1 - (1 - t)^2` over the 0→50% half, mirrored on 50→100%) | Visually identical to the chip-flash keyframe; recruiter cannot tell the SVG vs WebGL flash apart |
| Alpha curve | `1.0 → 0.5 → 1.0` ease-out (same curve as size) | Matches Phase 16 opacity keyframe |
| Halo overlay | No additional halo during flash — keep it crisp and short | Phase 16 flash is opacity+scale only; parity |
| Cleanup | `uFlashStartTime` resets to `-Infinity` after 100 ms so subsequent draws return to base state | Prevents stale flash on prop unchange |
| Reduced-motion | N/A — WebGL never mounts under reduced-motion | |

### 5. Renderer-swap transition

When the capability hook re-evaluates and crosses the 1024px breakpoint (resize OR override), the active renderer unmounts and the other mounts.

**Decision: instant snap. No crossfade.**

Rationale:
- Both renderers honor identical `layout` positions and `SKILL_CATEGORY_COLORS` — the user perceives "same constellation rendered differently," not "different scene."
- State (`selectedSkillId`, `filterState`, `highlightedSkillIds`) lives in `useConstellation` above the renderer slot, so swap preserves continuity automatically.
- Crossfade requires both renderers mounted simultaneously for 200ms — defeats the lazy-chunk benefit of WebGL and risks GPU contention.
- Instant swap matches Phase 16's bottom-sheet ↔ popover live-swap behavior (no crossfade there either).

| Event | Visual treatment |
|-------|-----------------|
| Viewport resize crosses 1024px upward | Capability hook re-evaluates → returns `'webgl'`. SvgConstellation unmounts. Suspense fallback shows SvgConstellation (same renderer) until WebGL chunk loads. Then WebGLConstellation mounts in its place — **one frame snap**. |
| Viewport resize crosses 1024px downward | Capability hook returns `'svg'`. WebGLConstellation unmounts (rAF loop cancelled, scene disposed via `geometry.dispose()` + `material.dispose()` + `renderer.dispose()`). SvgConstellation mounts — **one frame snap**. |
| URL override change (`?renderer=svg|webgl`) | Same as resize: one frame snap. |
| Theme change | NO swap; renderer stays mounted and updates uniforms (see §"Theme Reactivity"). |
| Language change | NO swap; WebGL has no text, only the surrounding DOM tree re-renders. |

**No animation library introduced.** All five animations are GLSL math in the existing `ShaderMaterial` fragment + vertex programs. Total animation shader budget: ≤80 LOC GLSL beyond the static rendering math.

## Loading State Contract

**Decision: SVG IS the loading state. No spinner. No skeleton.**

```jsx
{capability === 'webgl' ? (
  <RendererErrorBoundary>
    <Suspense fallback={<SvgConstellation {...sharedProps} />}>
      <WebGLConstellation {...sharedProps} />
    </Suspense>
  </RendererErrorBoundary>
) : (
  <SvgConstellation {...sharedProps} />
)}
```

| State | What renders | Duration | User perception |
|-------|-------------|----------|-----------------|
| Initial desktop visit, three.js chunk fetching | `SvgConstellation` (Suspense fallback) | ~50–300 ms on broadband; 1–3 s on slow desktop | "Constellation is alive immediately" — fallback IS the experience until WebGL replaces it |
| WebGL chunk loaded + scene initialized | `WebGLConstellation` swaps in | one rAF frame | One-frame snap to the WebGL render; both renderers honor identical positions/colors so the swap reads as "the same constellation continues" — no flicker, no reposition |
| WebGL chunk fetch fails (offline, blocked) | ErrorBoundary catches → `SvgConstellation` stays mounted | indefinite (until reload) | User never knows WebGL was attempted; no error UI |
| WebGL shader compile or runtime error post-mount | ErrorBoundary catches → swap to `SvgConstellation` (instant snap) | one rAF frame | Same — silent fallback |

**No new loading copy. No new error copy.** The `t.game.empty` and `t.game.error` strings from Phase 15 cover the only user-visible error path (graph data missing — same condition under both renderers). The renderer-fallback path is invisible to the user.

**Logging:** ErrorBoundary's `componentDidCatch` calls `console.error('[renderer-fallback]', error, errorInfo)`. No telemetry endpoint per D-17-ERRORBOUNDARY. Prefix `[renderer-fallback]` is greppable in browser DevTools for QA debugging.

## ErrorBoundary Fallback Visual Contract

Per D-17-ERRORBOUNDARY-INIT + D-17-ERRORBOUNDARY-RUNTIME:

| Caught error | Visual fallback | Console output |
|--------------|----------------|----------------|
| WebGL context creation failure post-feature-detect | `<SvgConstellation {...props}/>` instant snap | `console.error('[renderer-fallback] WebGL context init failed', error)` |
| Shader compile failure | `<SvgConstellation {...props}/>` instant snap | `console.error('[renderer-fallback] Shader compile failed', error)` |
| Lazy-chunk fetch network error | Suspense fallback (`<SvgConstellation/>`) remains; ErrorBoundary catches the chunk-load error and prevents re-attempt | `console.error('[renderer-fallback] WebGL chunk load failed', error)` |
| Runtime exception inside the rAF loop | Caught by boundary on next React render cycle; swap to SVG | `console.error('[renderer-fallback] WebGL runtime error', error)` |
| `webglcontextlost` event (canvas-level) | Canvas listener calls `event.preventDefault()`; on `webglcontextrestored` attempts re-init; on second failure within 5s, boundary fires SVG fallback | `console.warn('[renderer-fallback] WebGL context lost')` then `console.error` if recovery fails |

**Boundary state:** `{ hasError: boolean }`. Once `hasError === true`, the boundary does NOT re-attempt WebGL until page reload. There is NO "Try WebGL again" UI button — the user either reloads (and capability detection re-runs) or stays on the SVG path silently.

**No new copy strings.** The error/fallback path is invisible to the user; no `t.game.*` keys added.

## Focus indicator on WebGL

**The WebGL canvas itself never receives focus and never carries a focus ring.**

- Focus lives on the outer `role="application"` `<div>` wrapper (Phase 15) which holds `tabIndex={0}`.
- The wrapper's `focus-visible:ring-2 focus-visible:ring-brand` ring is the ONLY focus indication.
- Inside WebGL, "focused node" is a JS concept (the roving node id) used to draw the brand-color accent ring on the selected/focused node (see §Color "Accent reserved for" item 10) but is NOT exposed to the browser's focus engine.

**Why this matters for parity:**
- Phase 15 SVG path puts a brand-color `<circle>` ring around the focused SVG `<g>` node (the per-node focus ring). The Phase 15 keyboard handler (`spatialNav` + roving tabindex) updates which SVG node has `tabIndex={0}`.
- Phase 17 WebGL path CANNOT put per-node DOM focus rings (canvas has no DOM children). The shader-side brand-color ring on the selected node is the visual focus indicator under WebGL.
- **Keyboard navigation contract:** the roving-tabindex pattern from Phase 15 stays in the OUTER wrapper (managed by GameMode or by the Phase 15 SvgConstellation if it's the active renderer). Under WebGL, keyboard arrow events on the wrapper update a JS "roving node" state; the WebGL renderer reads that state via a prop (extend the Phase 15 contract with an optional `focusedNodeId` prop — currently `focusedNodeId` is internal to SvgConstellation; promote it to the shared contract OR pass via a new GameMode-owned context).

**Planner decision required:** does WebGL receive a `focusedNodeId` prop (preferred — explicit contract), or does the WebGL renderer also own the focus tracking internally? Recommendation: promote `focusedNodeId` to the shared renderer props contract; both renderers consume it; GameMode owns the roving state when WebGL is active.
