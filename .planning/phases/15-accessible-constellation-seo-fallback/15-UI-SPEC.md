# Phase 15 — UI Design Contract

> Visual and interaction contract for frontend phases. Generated by gsd-ui-researcher, verified by gsd-ui-checker.

---

## Scope of This Contract

Phase 15 ships the **SVG/DOM constellation renderer** behind the shared renderer props contract, the
**`GameMode` orchestrator** (capability detection, always resolves to SVG in this phase), and the
**sr-only/SEO full-experience fallback**. This is the gate-safe baseline that holds Lighthouse A11y=100 /
SEO=100 before the WebGL enhancement lands in Phase 17.

The established design system (Tailwind 3 + CSS-var tokens, dark/light themes, brand blue/emerald palette,
Inter + JetBrains Mono) is locked. This contract adds five net-new surfaces:

1. **H1/sub-copy block** — document heading above the constellation.
2. **SVG constellation canvas** — nodes, edges, halo, stagger-reveal, pulse.
3. **Hint pill** (reduced-motion fallback for the pulse affordance).
4. **`aria-live` announcement region** — polite SR announcements on selection.
5. **sr-only experience fallback** — full semantic list of all 12 experiences, always DOM-present.

### WCAG AA Contrast verification

**Node fill against the constellation backdrop** (dark `ink-800` = `#1A1A2E`):

| Node color | Hex | Contrast vs `#1A1A2E` | AA (4.5:1 text / 3:1 UI)? |
|------------|-----|----------------------|--------------------------|
| lang `#3b82f6` | blue-500 | ~4.6:1 | PASS (UI element ≥3:1) |
| ai `#a855f7` | purple-500 | ~5.1:1 | PASS |
| arch `#06b6d4` | cyan-500 | ~5.9:1 | PASS |
| cloud `#10b981` | emerald-500 | ~4.8:1 | PASS |
| devops `#f59e0b` | amber-500 | ~8.4:1 | PASS |
| security `#ef4444` | red-500 | ~4.1:1 | PASS (UI 3:1) |
| data `#8b5cf6` | violet-500 | ~4.7:1 | PASS |
| hardware `#ec4899` | pink-500 | ~4.5:1 | PASS |

**Node fill against light constellation backdrop** (light `ink-800` = `#F0F0F5`):

| Node color | Hex | Contrast vs `#F0F0F5` | AA? |
|------------|-----|----------------------|-----|
| lang `#3b82f6` | blue-500 | ~3.0:1 | BORDERLINE — use `fill-opacity: 1` + stroke `#2563EB` (blue-600, 4.97:1) to meet AA for UI components |
| ai `#a855f7` | purple-500 | ~3.5:1 | PASS (UI 3:1) |
| arch `#06b6d4` | cyan-500 | ~2.4:1 | FAIL on light — **apply `stroke="#0891b2"` (cyan-600, `#0891b2`, 3.6:1) as ring; fill opacity 0.85** |
| cloud `#10b981` | emerald-500 | ~2.9:1 | BORDERLINE — `stroke="#059669"` (emerald-600, 3.5:1) |
| devops `#f59e0b` | amber-500 | ~2.1:1 | FAIL on light — **`stroke="#d97706"` (amber-600) and add `stroke-width: 1.5` on light theme** |
| security `#ef4444` | red-500 | ~3.8:1 | PASS (UI 3:1) |
| data `#8b5cf6` | violet-500 | ~3.5:1 | PASS (UI 3:1) |
| hardware `#ec4899` | pink-500 | ~2.7:1 | BORDERLINE — `stroke="#db2777"` (pink-600, 3.2:1) |

**Light-theme AA fix pattern:** The SVG renderer MUST read the active theme and, when `theme === 'light'`,
apply a 1.5px colored stroke (one shade darker, from the fix table above) on each node circle. This brings
all failing/borderline nodes to ≥3:1 for UI components. The stroke does not replace the fill — it rings it.
The `--color-ink-800` light value (`#F0F0F5`) is the worst-case backdrop for node fill contrast.

**H1 / sub-copy text on page background:** `text-primary` dark (`#F0F0FF` on `#0D0D1A` = ~18:1 PASS) /
light (`#0D0D1A` on `#FFFFFF` = 21:1 PASS). No action needed.

**Hint pill text on pill background:** `text-secondary` dark (`#A0A0C0`) on `rgba(45,45,90,0.80)` → solid
equivalent `#3e3e72` → contrast ≈5.2:1 PASS. Light: `#525269` on `rgba(200,200,214,0.85)` → solid
`#cbcbd7` → contrast ≈4.8:1 PASS.

**Accent reserved for (explicit list — Phase 15):**
1. Focus-visible ring on the constellation root wrapper (`focus-visible:ring-2 focus-visible:ring-brand`).
2. Focus-visible ring on individual node wrappers (per-node `<g>` elements during tabfocus).
3. The active-node brightness-bump stroke on dark theme (a 2px `stroke` in `brand` color applied to the
   focused/selected node's circle element to complement the halo glow).
4. The `aria-live` announcement region — no visible accent, but the node label display (if shown on
   focus/hover) uses `text-brand` for the skill name text.

Accent is NOT used for node fills (those are locked semantic data colors), edge strokes, or the hint pill.

## Sizing Scale — Constellation Elements

### Node radius

Node radius formula: `r = FLOOR + (Math.sqrt(count) / Math.sqrt(MAX_COUNT)) * (CEIL - FLOOR)`

Where `MAX_COUNT` is the max `count` across all nodes (Java at count=11 in Phase 14 data).

**Mobile (viewport < 768px, SVG viewBox 0 0 1000 1000, rendered at ~360px):**

| Count | √count | Computed r (px in SVG units) | Visual radius (~device px at 360w) |
|-------|--------|-----------------------------|------------------------------------|
| 1 (min) | 1.0 | 6 | ~2.2px |
| 4 | 2.0 | 9 | ~3.2px |
| 9 | 3.0 | 12 | ~4.3px |
| 11 (Java, max) | 3.32 | 14 | ~5px |

**Floor: 6 SVG units. Ceiling: 14 SVG units (mobile). Ratio: 14/6 = 2.33×.**

**Desktop (viewport ≥ 768px, SVG viewBox 0 0 1000 1000, rendered at ~600–900px):**

| Count | Computed r (SVG units) | Visual radius (~device px at 800w) |
|-------|------------------------|-------------------------------------|
| 1 (min) | 10 | ~8px |
| 4 | 15 | ~12px |
| 9 | 20 | ~16px |
| 11 (Java, max) | 23 | ~18.4px |

**Floor: 10 SVG units. Ceiling: 23 SVG units (desktop). Ratio: 23/10 = 2.3×.**

_Note: The SVG uses a fixed viewBox `0 0 1000 1000` with `preserveAspectRatio="xMidYMid meet"`. The
layout bakes coordinates in 1000×1000 space. The renderer maps to actual pixels via `width/height`
attributes set to the container dimensions. The radius values above are **viewBox units**, not CSS px._

**Touch target for nodes (mobile):** The SVG `<circle>` hit area is too small for count=1 nodes (~2px
visual). Expand with a transparent `<circle r="22">` behind each node (44px diameter at the rendered
scale, matching WCAG 2.5.5 minimum). This invisible circle captures pointer/keyboard events but does not
affect visual.

### Edge stroke width

Edge stroke width uses `stroke-width` in SVG viewBox units:

| Weight | Stroke width (SVG units) | Visibility |
|--------|--------------------------|------------|
| 1 | 1.0 | Hidden by default (`opacity: 0`); revealed on hover/focus of incident node via CSS transition |
| 2 | 1.8 | Always visible |
| 3 | 2.8 | Always visible |
| 4+ | 4.0 | Always visible (cap at 4.0 to avoid visual noise) |

**Formula:** `strokeWidth = weight === 1 ? 1.0 : Math.min(1.0 + (weight - 1) * 1.0, 4.0)`

### Halo (active node outer glow)

The halo is a CSS `filter: drop-shadow(...)` applied to the active node's `<g>` element in SVG.
It is **not** a visible `<circle>` element — it uses SVG filter or CSS filter so it degrades gracefully.

| State | Filter value | Notes |
|-------|-------------|-------|
| Active (selected/focused) | `filter: drop-shadow(0 0 8px var(--color-constellation-halo)) drop-shadow(0 0 16px rgba(255,255,255,0.08))` | Two-pass: tight inner glow + soft outer glow |
| Hover (non-active) | `filter: drop-shadow(0 0 4px var(--color-constellation-halo))` | Single pass, half radius |
| Default | `filter: none` | |
| Reduced-motion | `filter: none` — halo suppressed entirely | Static brightness bump via `fill-opacity: 1.0` vs dim `0.35` only |

## Animation Contract

### Tailwind keyframes to add to `tailwind.config.js`

```js
// Add to theme.extend.keyframes:
nodeReveal: {
  '0%':   { opacity: '0', transform: 'scale(0.3)' },
  '70%':  { opacity: '1', transform: 'scale(1.1)' },
  '100%': { opacity: '1', transform: 'scale(1.0)' },
},
edgeReveal: {
  '0%':   { opacity: '0', strokeDashoffset: '100%' },
  '100%': { opacity: '1', strokeDashoffset: '0%' },
},
hintFadeOut: {
  '0%':   { opacity: '1' },
  '80%':  { opacity: '1' },
  '100%': { opacity: '0', pointerEvents: 'none' },
},

// Add to theme.extend.animation:
'node-reveal': 'nodeReveal 400ms cubic-bezier(0.34, 1.56, 0.64, 1) both',
'edge-reveal':  'edgeReveal 300ms ease-out both',
'hint-fade-out': 'hintFadeOut 600ms ease-in forwards',
// NOTE: pulse2 already exists in tailwind.config.js — reuse for node pulse
```

**Timing functions:**
- Node reveal: `cubic-bezier(0.34, 1.56, 0.64, 1)` — spring bounce (overshoot to 1.1× then settle). Teaches hierarchy: big nodes pop more.
- Edge reveal: `ease-out` — fast initial draw, taper at end. Draws from source to target using `strokeDasharray`/`strokeDashoffset`.
- Theme/language transition: `motion-safe:transition-colors duration-200` (existing global pattern).

### Reveal sequence (motion-safe path only)

```
Phase 1 — Node reveal (0ms → 800ms total):
  - Order: descending by count (Java first, then next biggest, ..., count=1 last)
  - Stagger: 30ms per node × 27 nodes = 810ms maximum stagger onset
  - Each node animation duration: 400ms
  - Last node completes at: 30ms × 26 + 400ms ≈ 1180ms
  - Implementation: inline style `animationDelay: {i * 30}ms` on each <g> element
    (i = index in the count-sorted node array, 0-indexed)

Phase 2 — Edge reveal (800ms → 1200ms):
  - All visible edges (weight ≥ 2) draw simultaneously after node reveal starts
  - Delay onset: 800ms (when large nodes are already visible)
  - Duration: 400ms each
  - Total complete: 1200ms

Total reveal: ~1.2s (matches D-15-LAND-PAINT)
```

### Pulse (motion-safe path only)

Reuse the existing `pulse2` keyframe (`0%,100%: opacity 1 → 50%: opacity 0.4`). Apply as
`motion-safe:animate-pulse2` on the Java node `<circle>` element (identified as the node with max count).

| Property | Value |
|----------|-------|
| Target | Biggest node by count (Java in current data) |
| Keyframe | `pulse2` (existing in tailwind.config.js) |
| Duration | 2s loop (`animate-pulse2` already configured with `2s ease-in-out infinite`) |
| Opacity range | 1.0 → 0.4 (from existing `pulse2` keyframe) |
| Class | `motion-safe:animate-pulse2` (MANDATORY — must not use bare `animate-pulse2`) |
| Stop condition | Remove the class (or set `animation: none`) on first interaction: hover enter on any node, keyboard focus moves into the constellation, or any node click |

### Reduced-motion path

The global `index.css` rule (`animation-duration: 0.01ms !important; animation-iteration-count: 1 !important`) suppresses all keyframe animations. The constellation must additionally:

1. Skip the stagger delay — render all nodes/edges at full `opacity: 1` from mount (no `animationDelay`).
2. Skip `animate-pulse2` — the `motion-safe:` prefix handles this.
3. Skip `filter: drop-shadow` halo — keep `filter: none` always (no performance cost, no distraction).
4. Show the **hint pill** instead of the pulse as the "start here" affordance.
5. Apply `fill-opacity` distinction immediately (active = 1.0, inactive = 0.35) without transition.

**Detection:** Use `window.matchMedia('(prefers-reduced-motion: reduce)').matches` in the component (or the existing `motion-safe:` Tailwind variant for class-based guards).

### Hint pill (reduced-motion + resting state)

The hint pill is DOM-present **only under `prefers-reduced-motion: reduce`**, or it appears as a
static overlay **after** the pulse stops on first interaction under the motion-safe path.

| Property | Value |
|----------|-------|
| Text | `t.game.hintPill` — EN: "Click a star · Toca una estrella" / ES: "Toca una estrella · Click a star" (bilingual even in a single string, per the design spec; the `·` separator needs no translation) |
| Position | Centered below constellation SVG, `mt-4` (16px margin-top) |
| Style | `bg-hint-pill-bg text-hint-pill-text text-xs font-mono rounded-full px-3 py-1.5 inline-block` |
| Aria | `aria-hidden="true"` — it is a visual affordance; SR users get the constellation `aria-label` instead |
| Fade-out | Applies `animate-hint-fade-out` + `pointer-events: none` on first interaction (same stop condition as pulse); removed from DOM after fade completes (`onAnimationEnd` → unmount) |
| Reduced-motion | Under `prefers-reduced-motion: reduce`: show statically, no fade animation. Remove from DOM on first interaction without animation. |

## ARIA Contract

### Constellation container role

**Role: `role="application"`**

Rationale: The constellation is a custom interactive widget with spatial arrow-key navigation that overrides
the browser's default arrow-key behavior (scrolling). WAI-ARIA APG defines `role="application"` for
"widgets that have their own keyboard shortcuts and do not follow typical widget interaction model." A
`role="grid"` would imply row/column structure that does not match the spatial layout. `role="listbox"`
implies linear single/multi selection. `role="tree"` implies hierarchy. `role="application"` correctly
signals to screen readers: "keyboard behavior is custom, follow the aria-label and embedded role semantics."

Per WAI-ARIA APG, `role="application"` regions **must** have an accessible name (via `aria-label` or
`aria-labelledby`) explaining the widget, and every focusable child must have a descriptive label.
Both are satisfied by this contract.

**Container element:**

```jsx
<div
  role="application"
  aria-label={t.game.constellationLabel}
  aria-roledescription={t.game.constellationRoleDesc}
  tabIndex={-1}              // container is not itself a tab stop; the roving node is
  ref={containerRef}         // used by Esc to return focus here, then blur to roving node
>
  <svg ...>
    {/* nodes rendered as <g> elements with role="button" */}
  </svg>
</div>
```

The outer wrapper that is the **single tab stop** in document flow is a sibling `<div>`:

```jsx
<div
  ref={rovingEntryRef}
  tabIndex={0}               // THE single tab stop — receives focus from Tab
  onKeyDown={handleKeyDown}
  className="focus:outline-none focus-visible:ring-2 focus-visible:ring-brand rounded-sm min-h-[44px]"
  aria-label={t.game.constellationLabel}
  role="application"
>
```

_Single element approach: the focusable `role="application"` div wraps the SVG. When Tab lands here,
keyboard focus is ON this element; arrow keys then move the roving `tabIndex=0` within the SVG nodes._

### Per-node ARIA

Each node is rendered as an SVG `<g>` element with these attributes:

```jsx
<g
  role="button"
  tabIndex={node.id === rovingNodeId ? 0 : -1}
  aria-label={buildNodeLabel(node, lang)}   // see template below
  aria-pressed={node.id === selectedSkillId}
  onClick={() => onSelectSkill(node.id)}
  onKeyDown={(e) => { if (e.key === 'Enter' || e.key === ' ') { e.preventDefault(); onSelectSkill(node.id) } }}
  ref={(el) => { nodeRefs.current[node.id] = el }}
  style={{ cursor: 'pointer' }}
>
  <circle r={node.r} cx={pos.x} cy={pos.y} fill={node.color} ... />
  <circle r={22} cx={pos.x} cy={pos.y} fill="transparent" />  {/* 44px touch target */}
</g>
```

**Node label template (`buildNodeLabel`):**

```js
// EN: "Java, Languages & Frameworks, used in 11 jobs."
// ES: "Java, Lenguajes & Frameworks, usado en 11 trabajos."
function buildNodeLabel(node, lang) {
  const categoryLabel = SKILL_CATEGORIES[node.category][lang]
  return lang === 'en'
    ? `${node.label}, ${categoryLabel}, used in ${node.count} ${node.count === 1 ? 'job' : 'jobs'}.`
    : `${node.label}, ${categoryLabel}, usado en ${node.count} ${node.count === 1 ? 'trabajo' : 'trabajos'}.`
}
```

This function is **not** a translation key — it is assembled from data at render time (category labels
already exist in `SKILL_CATEGORIES[k][lang]`). Add the count suffix strings to `translations.js`:

```js
// translations.js additions under game.*
game: {
  // existing keys ...
  nodeJobSingular: 'job',          // ES: 'trabajo'
  nodeJobPlural:   'jobs',         // ES: 'trabajos'
  nodeUsedIn:      'used in',      // ES: 'usado en'
}
```

### `aria-live` announcement region

```jsx
<div
  role="status"
  aria-live="polite"
  aria-atomic="true"
  className="sr-only"
>
  {announcement}
</div>
```

## Keyboard Navigation Contract

### Roving tabindex

- The constellation wrapper is **one Tab stop** in document order.
- Exactly one node has `tabIndex={0}` at any time: the **roving node** (`rovingNodeId` in state).
- All other nodes have `tabIndex={-1}`.
- Initial roving node: the node with the highest `count` (Java in current data).
- On Tab arrival: focus lands on the roving node entry element; the application `div` itself does not
  compete for Tab focus (it uses `tabIndex={-1}` or is managed to defer to the roving node).

### Arrow key spatial navigation

**Algorithm:** On `ArrowRight`, `ArrowLeft`, `ArrowUp`, `ArrowDown`:

1. Get the layout position `(x, y)` of the currently focused node from `computeLayout`.
2. For each other node, compute the dot product of the direction vector `(dx, dy)` with the arrow vector:
   - `ArrowRight`: vec = `(1, 0)` — find min distance among nodes where `(nx - cx) * 1 > 0`
   - `ArrowLeft`:  vec = `(-1, 0)` — find min distance among nodes where `(nx - cx) * -1 > 0`
   - `ArrowDown`:  vec = `(0, 1)` — find min distance among nodes where `(ny - cy) * 1 > 0`
   - `ArrowUp`:    vec = `(0, -1)` — find min distance among nodes where `(ny - cy) * -1 > 0`
3. Among qualifying nodes (those in the correct half-plane), pick the one with the smallest Euclidean
   distance to the current node.
4. Set that node as `rovingNodeId`, move `tabIndex={0}` to it, call `.focus()` on its ref.
5. If no node qualifies in the arrow direction (current node is at an edge), wrap to the farthest node
   in the opposite half-plane (the most extreme in the given direction — treats the canvas as circular).
6. Call `e.preventDefault()` on all arrow key events to prevent page scrolling.

**Key bindings summary:**

| Key | Action |
|-----|--------|
| `Tab` | Moves focus INTO the constellation (to roving node) or OUT of it (standard Tab behavior) |
| `ArrowRight` / `ArrowLeft` / `ArrowUp` / `ArrowDown` | Spatial jump to nearest node in direction |
| `Enter` / `Space` | Activate focused node (`onSelectSkill(rovingNodeId)`) |
| `Esc` | Clear selection (`onSelectSkill(null)`); return focus to constellation root (call `rovingEntryRef.current.focus()`) |
| `Tab` while inside | Move out of constellation to next document focus stop |

### Focus indicator

Matching `ThemeToggle` and `ViewModeToggle` patterns:

```
focus:outline-none focus-visible:ring-2 focus-visible:ring-brand
```

On the SVG `<g>` nodes: since SVG does not natively support CSS `outline`, implement the focus ring as a
`<circle>` ring element that appears on `:focus-visible`:

```jsx
// Inside the node <g>, conditionally rendered when node has tabIndex={0} and is document-focused:
{isFocused && (
  <circle
    cx={pos.x}
    cy={pos.y}
    r={nodeRadius + 4}       // 4px outside the node fill
    fill="none"
    stroke="var(--color-brand)"
    strokeWidth="2"
    className="pointer-events-none"
    aria-hidden="true"
  />
)}
```

The focus detection uses a `focusedNodeId` state (updated via `onFocus`/`onBlur` on each `<g>`).

and avoids browsers that expose SVG `<g>` inconsistently. The executor may choose either; the ARIA
contract's requirements (per-node label, live region, Esc behavior) must hold in both approaches._

## sr-only Fallback Contract

### Language policy decision (D-15-SEO-DEFAULTS + SEO research)

**Decision: active-lang only.**

Rationale: Google reads **visible content** for language detection, not `sr-only` hidden text (confirmed by
Google Search Central documentation — Google does not use hidden text for language classification in its
algorithm). The sr-only fallback's primary value is:
1. Screen reader users (announced in active language — the language they've chosen).
2. ATS resume parsers (which may parse DOM text; active-lang is the relevant CV language for the submission).
3. Secondary crawler content (Googlebot indexes it as supplementary content in the active language).

Rendering both languages in sr-only adds ~4KB DOM weight per language toggle without SEO gain (Google
would index a mixed-language block as ambiguous). Active-lang only is the correct call.

The language toggle re-renders the full React tree; the fallback automatically switches language when
`lang` changes.

### Fallback structure

```jsx
// src/game/ConstellationFallback.js
export default function ConstellationFallback({ experiences, lang }) {
  return (
    <section
      aria-labelledby="constellation-fallback-heading"
      className="sr-only"
    >
      <h2 id="constellation-fallback-heading">
        {t.game.fallbackHeading}
        {/* EN: "Full career experience" / ES: "Experiencia profesional completa" */}
      </h2>
      <ol>
        {experiences.map((exp, i) => (
          <li key={i}>
            <article>
              <h3>{exp.title[lang]} — {exp.company}</h3>
              <p>{exp.date} · {exp.location}</p>
              <ul>
                {exp.bullets[lang].map((b, j) => (
                  <li key={j}>{b}</li>
                ))}
              </ul>
              <p>
                {/* tech chips as plain text */}
                {exp.tech.join(', ')}
              </p>
            </article>
          </li>
        ))}
      </ol>
    </section>
  )
}
```

**Structure requirements:**
- `<section>` with `aria-labelledby` pointing to an `<h2>` — creates a labeled landmark.
- Ordered `<ol>` — chronological order, most recent first (matching Experience.js order in `experience.js`).
- Each entry: `<article>` with `<h3>` for title+company, date+location as `<p>`, bullets as `<ul>`, tech as
  `<p>` plain text (comma-separated).
- Uses Tailwind `sr-only` utility (NOT `visibility: hidden` or `display: none` — those hide from SR).
- Component receives `experiences` (the raw `EXPERIENCE` array) and `lang` as props — no hook calls;
  pure presentational component, easy to test.
- **Always in the DOM** regardless of render path (SVG or future WebGL). `ConstellationFallback` is
  rendered by `GameMode`, not by `SvgConstellation`.
