<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| npm scripts → shell environment | `$LIGHTHOUSE_TARGET_URL` injected from user shell; quoting must prevent command injection if URL contains shell metacharacters |
| Lighthouse report file → check script | `lighthouse:check` reads JSON written by `lighthouse`; if file missing or malformed, script must fail cleanly not silently |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-11-02-01 | E (Elevation) | `$LIGHTHOUSE_TARGET_URL` shell expansion in npm scripts | mitigate | Variable is double-quoted in both `lighthouse:mobile` and `lighthouse:deployed` (`"$LIGHTHOUSE_TARGET_URL"`). User control of this var is acceptable — it's run from their own shell, not an external attacker. Lighthouse itself validates the URL format. |
| T-11-02-02 | T (Tampering) | `lighthouse-report-mobile.report.json` (between mobile + check steps) | accept | Both scripts run on the same machine in sequence; file system is the trust boundary. If a local attacker has write access to the working dir between `lighthouse:deployed` and `lighthouse:check`, they have full code execution anyway. |
| T-11-02-03 | D (DoS) | Lighthouse subprocess (headless Chrome) | accept | Chrome flags include `--no-sandbox --disable-gpu` — appropriate for headless CI/CLI. No public-facing service; user-initiated. |
| T-11-02-04 | I (Info Disclosure) | Lighthouse report HTML committed to repo (Plan 11-05) | accept | Report contains the audited URL + page structure. Phase 11 URL is `*.vercel.app` — already public. No secrets or auth headers logged. |

All threats LOW. No new attack surface added; this plan only modifies build-tooling scripts in a developer-controlled environment.
