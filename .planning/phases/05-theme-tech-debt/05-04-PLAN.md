WCAG AA thresholds:
- Normal text (< 18pt regular or < 14pt bold): 4.5:1 minimum
- Large text (>= 18pt regular or >= 14pt bold): 3:1 minimum
- UI components (buttons, form controls): 3:1 minimum
- Decorative or disabled elements: no minimum

Computed contrast ratios (sanity check before audit):
- #0D0D1A on #FFFFFF: ~19.5:1 (passes AA easily)
- #525269 on #F5F5FA: ~7.5:1 (passes AA)
- #7A7A92 on #FFFFFF: ~4.6:1 (BORDERLINE — may fail in some scenarios)
- #6C63FF on #FFFFFF: ~4.8:1 (passes AA normal text)
- #6C63FF on #F5F5FA: ~4.6:1 (BORDERLINE)
- #E64444 on #FFFFFF: ~4.5:1 (BORDERLINE)

      --only-categories=accessibility \

    Practical approach: Use Puppeteer to launch Chrome, set localStorage, then run Lighthouse via puppeteer-controlled Chrome. OR simplest: clear a separate Chrome profile, navigate to site, click theme toggle to light, run Lighthouse from DevTools.

  <action>
    CONDITIONAL TASK — only execute if Task 1 found violations.

    For each contrast violation:
    1. Identify the CSS variable causing the failure
    2. Compute a darkened value that hits the required ratio
    3. Update the variable inside [data-theme="light"] block in src/index.css

    Likely candidates and pre-computed alternatives if violations occur:

    **If --color-text-muted (#7A7A92) fails on white:**
    - Replace with: #6B6B85 (darker, ratio ~5.2:1 on #FFFFFF)
    - Or further: #5C5C75 (ratio ~6.0:1)

    **If --color-brand (#6C63FF) fails as button text or link on surface-card:**
    - Replace with: #5B53E8 (ratio ~5.6:1 on #F5F5FA)
    - Use --color-brand-strong as the visible token if needed; light-mode override can swap them

    **If --color-accent-dark (#E64444) fails:**
    - Replace with: #D73838 (ratio ~5.0:1 on #FFFFFF)

    For ANY chip/badge backgrounds that fail (e.g., low contrast tag pills), consider lightening the background OR darkening the text:
    - Lighten background: e.g., from #EBEBF0 → #DEDEE5 (more contrast vs dark text)
    - Darken text: stick with #0D0D1A (already maximum)

    DO NOT:
    - Modify the dark mode :root block
    - Introduce new variable names
    - Touch tailwind.config.js
    - Touch any component's Tailwind classes

    DO:
    - Update only the value(s) inside [data-theme="light"] for the failing variable(s)
    - Re-run Lighthouse after each fix to confirm regression closed

    If NO violations found in Task 1: skip this task entirely. Note in summary "no edits required, light tokens passed AA on first audit".

    Implements VIS-01 (success criterion #3: light mode passes WCAG AA) per phase requirements.
