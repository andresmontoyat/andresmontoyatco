<objective>
Code-split the production bundle so the initial JS payload covers only above-fold content, with below-fold sections loaded on demand.

**D-02 (code splitting per Discretion at CONTEXT.md line 60):** Wrap Experience, Contact, and Footer in `React.lazy + Suspense`. These three sections are visible only after the user scrolls past the Hero + About + Skill (above-fold + first scroll). Splitting them creates 3 additional chunks that fetch in parallel as the browser idles after first paint.

Why these three (not About / Skill):
- **Hero** — hero/initial paint, cannot lazy.
- **About** — appears immediately after Hero on a typical viewport. Lazy would cause a Suspense flash on slow networks.
- **Skill** — same; second visible section. Lazy boundary noise > bundle-size savings.
- **Experience** — large component (113 lines + EXPERIENCE data import = 12 entries × bilingual content). Best ROI for lazy.
- **Contact** — moderate size, includes EmailHeroCard logic + clipboard handler. Good lazy candidate.
- **Footer** — small but always below scroll, no SEO/LCP cost to lazy.

**Suspense fallback strategy:** A single shared minimal fallback (`<div className="py-20" aria-hidden="true" />`) preserving the section's vertical rhythm so layout doesn't shift when chunks land. The 4-color brand spinner / loading states are out of scope — Phase 4 is polish, not new UI.

Acknowledged trade-off (CONTEXT.md line 25): "limited gain on SPA but every byte helps mobile." This plan is targeted at Lighthouse's "Reduce unused JavaScript" + "Reduce initial server response time for hostnames" sub-checks.

Plan 04-02 already produced `dist/stats.html`. Before adding lazy boundaries, this plan reads the visualizer to confirm the 3 split candidates are non-trivial in size (sanity check: if Footer is 200 bytes, splitting wastes a chunk-fetch round trip).

Purpose: Reduce Time-To-Interactive (TTI) on mobile by deferring below-fold JS.
Output: 1 file modified (src/App.js); 3 lazy chunks emitted in dist/assets/; build clean.
</objective>

<interfaces>
<!-- React.lazy + Suspense pattern — React 18 stable -->

API:
```jsx
import React, { Suspense } from 'react'

const Experience = React.lazy(() => import('./components/Experience'))
const Contact    = React.lazy(() => import('./components/Contact'))
const Footer     = React.lazy(() => import('./components/Footer'))

// Inside JSX tree:
<Suspense fallback={<div className="py-20" aria-hidden="true" />}>
  <Experience />
</Suspense>
```

Key constraints:
- The file passed to `import('./components/X')` must default-export X. All three target components do (verified via reading the files).
- Suspense boundary scope: one Suspense per lazy section keeps the fallback localized to that section's vertical region. A single Suspense wrapping all three would block until ALL three chunks load — slower TTI.
- Fallback must take vertical space matching the lazy component's `py-20` (or comparable) so CLS stays 0 when the chunk lands. Otherwise the page reflows and Lighthouse flags CLS.
- During reduce-motion mode, the fallback transition is fine — no animation involved (just a div placeholder).

Vite Rollup behavior: `import('./components/X')` produces a separate chunk in dist/assets/X-{hash}.js. Vite emits the chunk's preload links automatically when the parent JS loads, so the chunk fetches in parallel with the user's first scroll.

    **Step 1 — Modified App.js (full file replacement):**

    ```jsx
    import React, { Suspense } from 'react'

    import { LanguageProvider, useLanguage } from './i18n/LanguageContext'
    import Nav from './components/Nav'
    import Hero from './components/Hero'
    import About from './components/About'
    import Skill from './components/Skill'

    const Experience = React.lazy(() => import('./components/Experience'))
    const Contact = React.lazy(() => import('./components/Contact'))
    const Footer = React.lazy(() => import('./components/Footer'))

    const SectionFallback = (
      <div className="py-20" aria-hidden="true" />
    )

    function SkipLink() {
      const { t } = useLanguage()
      return (
        <a href="#main" className="skip-link">
          {t.nav.skipToContent}
        </a>
      )
    }

    export default function App() {
      return (
        <LanguageProvider>
          <div className="min-h-screen bg-ink-900 text-text-primary font-sans antialiased bg-hero-gradient bg-grid-subtle">
            <SkipLink />
            <Nav />
            <main id="main">
              <Hero />
              <About />
              <Skill />
              <Suspense fallback={SectionFallback}>
                <Experience />
              </Suspense>
              <Suspense fallback={SectionFallback}>
                <Contact />
              </Suspense>
            </main>
            <Suspense fallback={SectionFallback}>
              <Footer />
            </Suspense>
          </div>
        </LanguageProvider>
      )
    }
    ```

    **Step 2 — Key invariants:**
    - `Suspense` is imported as a named export from `react` (alongside default React import).
    - 3 separate `<Suspense>` wrappers, NOT one wrapping all three — keeps fallbacks localized so a slow Experience chunk doesn't hold up Contact's render.
    - Footer's `<Suspense>` is OUTSIDE `<main>` — Footer is a `<footer>` landmark, not a `<main>` child. Preserves landmark semantics.
    - `SectionFallback` is a module-level const (not a component instance per render) — single shared JSX element saves a few render cycles.
    - The static eager imports for Hero/About/Skill stay — they're above-fold and need to be in the initial bundle.
    - `LanguageProvider` is the outer wrapper; both eager and lazy components consume `useLanguage` from inside it.

    **Step 3 — Why `aria-hidden="true"` on the fallback:** Screen readers should not announce "loading region" for what is essentially a layout placeholder. The actual content arrives within ~50ms on broadband, ~500ms on 3G — fast enough that the fallback is visual rhythm only.

    **Step 4 — Reduce-motion compatibility:** The fallback `<div>` has no animation. The lazy components themselves use `motion-safe:` prefixes per CLAUDE.md. No conflict.
