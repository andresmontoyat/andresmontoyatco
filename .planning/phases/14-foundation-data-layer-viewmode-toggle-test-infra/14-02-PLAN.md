<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| URL query string (`?mode=`) → client state | The only externally-controllable input in this plan; an attacker can put any value in the `mode` param |
| localStorage → client state | Persisted value re-read on load (first-party origin only) |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-14-02 | Tampering / Injection | `?mode=` parse in ViewModeContext.readInitialMode | mitigate | Validate the param strictly against the `['game','dev']` allowlist; only an exact allow-listed value is ever assigned to state. The raw param is never reflected into the DOM, used as a key, or interpolated anywhere. Unit test asserts `?mode=hacker` is ignored. |
| T-14-03 | Tampering | localStorage `cam-viewmode` value | mitigate | Stored value is re-validated against VALID_MODES on read; an out-of-allowlist stored value falls back to default. |
| T-14-01 | Denial of Service | localStorage unavailable / throws (private mode) | mitigate | Every storage touch is wrapped in try/catch with a `typeof window` guard; on failure the app defaults to game mode in memory and never crashes (mirrors LanguageContext/ThemeContext). |
| T-14-XSS | Cross-Site Scripting | placeholder copy render | accept | Placeholder text comes from the static first-party `translations.js` dictionary rendered as React text nodes (auto-escaped); no `dangerouslySetInnerHTML` is introduced. No untrusted data reaches the DOM. |

No new dependencies are installed in this plan (test infra came from Plan 14-01), so there is no supply-chain install surface here. Overall surface remains minimal (client-only static SPA). Block only on a high-severity finding (none expected); the `?mode=` allowlist closes the one real input boundary.
</threat_model>
